# document-generator 仕様書

リフォレの仲介案件における書類作成業務を、フォーム入力からPDF出力まで自動化するシステム。
**1 つの案件マスタシートに全案件・全情報を集約し、月末に複数案件をまとめて書類出力する** 集約型・バッチ型運用が中核。

- **対象 GAS 種別**: Sheets コンテナバインド型（案件マスタシート側）+ Forms トリガー（Phase 4 以降）
- **タイムゾーン**: `Asia/Tokyo`
- **GAS ランタイム**: V8
- **設計版**: 2026-05-27 改訂（旧「1案件=1スプレッドシート」方式から「1案件=1行+1フォルダ」方式に変更。詳細は本書末尾「設計変更履歴」参照）

---

## 1. North Star（目標形）

```
┌─────────────────────────────────────────────────────────┐
│ ① Google フォーム                                          │
│   - 売主情報、買主情報、土地情報、立木情報、金額条件 等  │
│   - 案件パターン（A①/A②/A③/B①/B②）を選択                  │
│   - 登記簿PDFアップロード → OCR で自動入力（Phase 5）       │
└──────────────────────┬──────────────────────────────────┘
                       │ onFormSubmit トリガー
                       ↓
┌─────────────────────────────────────────────────────────┐
│ ② 案件マスタ（1シート集約型・1案件=1行）                  │
│   - 案件IDを自動採番（EST-YYYYMMDD-001）                   │
│   - フォーム入力値を1行追加                                │
│   - 案件パターン → 書類列に初期チェック投入                │
│   - 契約日から /案件/YYYY/MM/ フォルダ作成                 │
│                                                            │
│   列構成:                                                  │
│   [選択☑] [案件ID] [案件情報...] [書類1☑] [書類2☑]...     │
│     ↑ 行頭チェック        ↑ 書類ごとにチェックボックス      │
│   [ステータス▼] [フォルダURL] [更新日時]                  │
└──────────────────────┬──────────────────────────────────┘
                       │ 営業/事務が必要に応じて編集
                       │ 月末に複数行を選択
                       │ メニュー「書類発行」クリック
                       ↓
┌─────────────────────────────────────────────────────────┐
│ ③ 一括 PDF 出力（GAS バッチ処理）                          │
│   - 行頭チェック ON の案件を抽出                           │
│   - 各案件について、書類列チェック ON のものを順次 PDF 化  │
│   - 案件フォルダに保存（再出力時は v2, v3... を付与）      │
│   - 完了後、ステータスを「出力済み」に更新                  │
│   - 完成リストをダイアログに表示（フォルダリンク付き）      │
└─────────────────────────────────────────────────────────┘
```

    │                                   # フォーム回答も直接ここに追記
    ├── 集計/
    │   ├── 2026年05月チェックリスト.gsheet  # 月次レポート（Phase 6 で自動生成）
    │   └── ...

    │   ├── お客様契約書_土地込.gdoc
    │   └── ...
    └── 案件/
        ├── 2026/
        │   ├── 05/
        │   │   ├── EST-20260517-001_鹿屋_佐藤てるお/

        │   │   │   ├── EST-20260517-001_お客様請求書_20260517.pdf
        │   │   │   └── ...
        │   │   └── EST-20260520-001_南さつま_山田太郎/
        │   │       └── ...

**ポイント**:
- **案件マスタ 1 シートに全案件を集約**（旧設計の「1案件=1スプレッドシート」は廃止）
- **`案件/YYYY/MM/<案件ID>_<地域>_<相手方名>/`** で物理的に月別に区切る（目視管理・アーカイブ容易）
- **相手方名はパターン依存**: A 系統なら売主名義人、B 系統なら買主名義人
- 月次集計は **案件マスタを QUERY / SUMIFS するだけ** で成立（冗長転記レイヤなし）

## 2. 業務フローの理解

```
[地主] ─(A: 仕入)─→ [リフォレ] ─(B: 売上)─→ [買主 / 伐採業者]
```

契約書は A 系統（リフォレが乙＝買主）と B 系統（リフォレが甲＝売主）の2系統。
さらに「土地が絡むか / 立木のみか」で分岐。

| 系統 | パターン | 対象 | 契約書 | 必要なその他書類 |
|---|---|---|---|---|
| A | A① | 立木のみ | 立木売買契約書 | — |
| A | A② | 立木+土地 | 立木付土地売買契約書 | 法務局関連 5 種 |
| A | A③ | 土地のみ | 土地売買契約書 | 法務局関連 5 種 |
| B | B① | 立木のみ | お客様契約書_立木 | — |
| B | B② | 立木+土地（A③ 土地のみケースも含む） | お客様契約書_土地込 | 法務局関連 5 種 |

土地が絡む場合（A②/A③/B②）は法務局関連書類 5 種類が追加で必要。
売上系（B）は見積書・請求書・売上計算シートが追加で発生する。

## 3. 段階的アプローチ（Phase 設計）

各 Phase は **使える状態で止める**。Phase 完了時に事務担当が一部でも使える価値を提供する。

| Phase | ゴール | 状態 |
|---|---|---|
| **0** | 情報収集・仕様化 | ✅ 完了 |
| **1** | 既存「案件一覧」シートからアクティブ行を選び、「土地売買契約書」1書類を PDF 出力する MVP（**現状の単発生成フロー**） | 🚧 進行中（ユーザー作業待ち） |
| **2** | **新方式: 案件マスタ集約型 + 複数案件×複数書類の一括出力**。書類列チェックボックス、行頭選択チェックボックス、パターン→書類セット自動チェック、再出力時のバージョニング | 未着手 |
| **3** | 明細繰返し対応、A①/A②/B①/B② の 3 契約書実装、法務局関連 5 書類 | 未着手 |
| **4** | Google フォーム連携（`onFormSubmit` → 案件 ID 採番 → 案件マスタ追記 → 案件フォルダ作成） | 未着手 |
| **5** | フォームに登記簿 PDF 添付欄追加、registry-ocr と統合 | 未着手 |
| **6** | **月次チェックリスト自動生成**（案件マスタからの集計レポート）、見積書/請求書/売上計算シート対応 | 未着手 |
| **7+** | 運用フィードバックを反映した改善 | 未着手 |

> **重要**: Phase 1 は現状の「案件一覧」シートをそのまま使い、1 書類だけ動かす最小実装。
> Phase 2 で本格的なバッチ型・集約型運用にシフトする。マスタ列追加・チェックボックス UI 化・パターン判定はすべて Phase 2。

### Phase 1 詳細（最初の MVP）

**ゴール**: 既存のスプレッドシート（[【リフォレ】書類自動生成](https://docs.google.com/spreadsheets/d/12BEUxTqnADqWsmffGWos0tFe08FohEE-0RQJbLNAnEk/edit)）の案件一覧シートからアクティブ行を選び、土地売買契約書 1 書類を PDF 出力できる状態を作る。

**手順**:
1. 「土地売買契約書」テンプレ（[fileId: 1AsZLK9_rmIF-VC62I42QMAmPmzNz0kp1KmegOZ6ApJI](https://docs.google.com/document/d/1AsZLK9_rmIF-VC62I42QMAmPmzNz0kp1KmegOZ6ApJI/edit)）に `{{key}}` プレースホルダを挿入
2. 案件一覧シートから案件 ID をキーに該当行のデータを取得
3. プレースホルダ置換 → 指定フォルダに PDF 保存
4. メニュー「書類出力」→「土地売買契約書」項目で実行
5. `docs/documents/purchase_contract_land.md` を仕様書として完成

**Phase 1 で意図的にやらないこと**:
- 複数案件×複数書類の一括処理 → Phase 2
- 行頭チェックボックスによる選択 → Phase 2
- 書類列チェックボックス UI → Phase 2
- パターン判定 → Phase 2
- 明細表の繰返し処理 → Phase 3
- フォーム連携 → Phase 4
- OCR 統合 → Phase 5

### Phase 2 詳細（バッチ型・集約型運用の本格導入）

**ゴール**: 案件マスタ 1 シートに全案件を集約し、月末に複数案件×複数書類をまとめて出力できる状態。**この Phase が新 North Star の中核**。

**マスタシート再設計**:
- 行頭に **選択チェックボックス列** を追加（出力対象選択用）
- **案件パターン列** を追加（プルダウン: A①/A②/A③/B①/B②）
- **売主名義人・買主名義人** を別列に保持（系統依存）
- **書類ごとにチェックボックス列** を追加（14 書類分。詳細は [`master-sheet-schema.md`](./master-sheet-schema.md)）
- **ステータス列** をプルダウン化（未着手 / 出力済み / 完了 / 保留）
- **フォルダURL列** を自動セット

**書類マスタシート新設**:
- 書類 ID、書類名、テンプレ ID、対応パターン、命名規則、明細含むか
- 詳細は [`master-sheet-schema.md`](./master-sheet-schema.md)
- 「出力書類リスト」シート（人間運用）から手動で初期データ転記

**取引先マスタ・代理人マスタ新設**:
- 伐採業者・買主など法人 → 取引先マスタ（[`master-sheet-schema.md` § 9](./master-sheet-schema.md#9-取引先マスタシートphase-2-新設)）
- 法務局申請書の代理人 → 代理人マスタ（[§ 10](./master-sheet-schema.md#10-代理人マスタシートphase-2-新設)）
- 案件マスタに `買主取引先ID` / `売主取引先ID` / `代理人ID` 列を追加し、プルダウン参照
- 売主（個人地主）は案件マスタに直接記載（マスタ化しない）
- 自社（リフォレ）情報はテンプレに固定文字列で残す（マスタ化しない）

**GAS 機能追加**:
- メニュー「書類発行」→ 選択行を抽出 → 各行の ON 書類を順次生成
- パターン選択時に該当書類列の初期チェックを自動投入（onEdit トリガー）
- 案件フォルダ未作成なら作成 → URL をマスタに書き戻し
- 既存 PDF が同名にあれば `_v2`, `_v3` ... と suffix
- 完了後にステータスを「出力済み」に更新（手動で「完了」に変更可能）
- 進捗・結果をダイアログに表示

### Phase 3 詳細（明細繰返し + 残契約書）

**ゴール**: 物件明細を表として差し込む処理を実装。A①/A②/B①/B② 4 契約書と法務局関連 5 書類を実装。

**追加実装**:
- プレースホルダ規約に `{{#each items}}...{{/each}}` 構文を実装
- 物件シートのデータを案件 ID で集約 → 明細表として差し込み
- 書類マスタの「明細含む」フラグで分岐
- 法務局関連 5 書類（土地絡みの場合のみ自動 ON）

### Phase 4 詳細（Google フォーム連携）

**ゴール**: 新規案件を Google フォームから入力。送信トリガーで案件マスタに 1 行追加され、案件フォルダが Drive に作成される。

**追加実装**:
- Google フォームを作成（案件パターン分岐込み、複雑な条件分岐は GAS 側でカバー）
- フォーム送信トリガー (`onFormSubmit`) で:
  - 案件 ID 採番（`EST-YYYYMMDD-連番3桁`）
  - 契約日から `案件/YYYY/MM/<案件ID>_<地域>_<相手方名>/` フォルダ作成
  - 案件マスタに 1 行追記（パターン列 → 書類列チェック初期投入）
  - フォルダ URL をマスタにセット
- フォーム回答シート（Google Forms が作るシート）はマスタとは別に保持。マスタ追記処理が冪等になるよう設計

**フォーム vs 自作の判断**:
- Phase 1〜3 は Google フォーム未使用（既存シートを直接編集）
- Phase 4 で Google フォーム導入（UI シンプル・運用負荷低・スマホ対応完璧）
- 自作フォーム（Next.js 等）は **将来要件が固まってから判断**。当面は Google フォームで足りる

### Phase 5 詳細（OCR 統合）

**ゴール**: フォームに登記簿 PDF 添付欄を作り、送信時に registry-ocr のロジックで自動抽出。結果を案件マスタに自動投入。

**追加実装**:
- フォームに `登記簿PDF` 添付欄追加
- フォーム送信トリガーで添付 PDF を取得 → registry-ocr の OCR ロジックを呼び出し
- 抽出結果（所在・地番・地目・地積・所有者氏名）を案件マスタの該当列に投入
- registry-ocr 側のコードを `shared/` などに切り出し、両プロジェクトで共有する設計に変更検討

### Phase 6 詳細（月次チェックリスト + 残り書類）

**ゴール**: 案件マスタを集計基盤として月次チェックリストを自動生成。あわせて見積書・請求書・売上計算シートに対応。

**月次チェックリストの自動生成**:
- 集計テンプレートシート（[既存サンプル](https://docs.google.com/spreadsheets/d/126zJY6MttfzXMYnRmxlUkAOMy6f_HQwrc27Sz8F82DY/edit)）を `テンプレート/チェックリスト.gsheet` として整備
- GAS メニュー「月次レポート生成」または定期トリガーで:
  - 対象年月を指定
  - テンプレを `集計/` 配下にコピー（命名: `YYYY年MM月チェックリスト.gsheet`）
  - 案件マスタから対象月の行を `QUERY` で抽出 → 担当営業ごとにグルーピング
  - 「未契約合計」「契約済合計」「担当別合計」「全体合計」を自動算出

**追加書類対応**:
- 見積書: 既存の `generateEstimatePdfFromActiveRow` を置き換え、Sheets テンプレから PDF 化
- 請求書: 同じく Sheets ベース
- 売上計算シート / チェックリストは集計基盤の上に乗る形で実装

## 4. 入出力仕様

### 4.1 入力

詳細は [`master-sheet-schema.md`](./master-sheet-schema.md) を参照。

- **Phase 1**: 既存スプレッドシート [【リフォレ】書類自動生成](https://docs.google.com/spreadsheets/d/12BEUxTqnADqWsmffGWos0tFe08FohEE-0RQJbLNAnEk/edit) の「案件一覧」「物件」シート（アクティブ行 1 件）
- **Phase 2〜**: 同シートを **「案件マスタ」** に列拡張。複数案件×複数書類を行頭・書類列チェックボックスで指定
- **Phase 4〜**: Google フォーム → `onFormSubmit` → 案件マスタに自動追記

### 4.2 出力（PDF）

| 項目 | 内容 |
|---|---|
| ファイル形式 | PDF |
| 保存先 | 案件フォルダ（Phase 1 では設定シートで指定、Phase 2 以降は `案件/YYYY/MM/<案件ID>_<地域>_<相手方名>/` に自動配置） |
| ファイル名 | `{案件ID}_{書類名}_{YYYYMMDD}.pdf`（再出力時は `_v2`, `_v3` ... を付与） |

### 4.3 ステータス遷移

| ステータス | 意味 |
|---|---|
| `未着手` | フォーム入力直後、または手動追加直後の初期値 |
| `出力済み` | 「書類発行」メニューで PDF が生成された直後（自動セット） |
| `完了` | 送付・契約締結など最終状態（手動セット） |
| `保留` | 何らかの理由で進行を止めている状態（手動セット） |

「出力済み」案件を再度「書類発行」メニューにかけると、PDF ファイル名に `_v2` 等の suffix が付き、ステータスは「出力済み」のまま（再出力扱い）。

## 8. エラーハンドリング方針

- マスタに必須値が無い → 該当案件のみスキップ + ログに警告（バッチ全体は止めない）
- テンプレのプレースホルダがマスタに無い → 警告ログ + 処理継続（未差込のまま PDF 出力）
- 既存 PDF と同名 → `_v2`, `_v3` ... と suffix（上書きしない）
- フォルダ作成失敗 → 該当案件のみスキップ + ログに警告
- 一時 Docs はエラー時も `finally` で削除
- バッチ完了後にダイアログで「成功 X 件 / 失敗 Y 件 / 詳細リスト」を表示

### 2026-05-27: 出力書類リストのスプレッドシート移行と取引先マスタ導入

**変更内容**:
1. 出力書類リスト（14 書類の【原本】【テンプレ】Docs URL）を `documents/README.md` の表からスプレッドシート内の「出力書類リスト」シートに移行。
2. 取引先マスタ・代理人マスタを Phase 2 で新設することを決定（伐採業者・買主など法人と、法務局申請書の代理人をマスタ化）。
3. 売主（個人地主）情報は案件マスタに直接記載（マスタ化しない）。
4. 自社（リフォレ）情報はテンプレに固定文字列で残す（マスタ化しない）。

**背景**:
- 【原本】テンプレに「ほぼこの会社／代理人」という慣行で印字されていた固定取引先情報が、案件によって差し替えが必要なケースがあると判明。
- 紙運用時代の固定印字をそのままシステム化すると差し替えが手作業になるため、マスタ参照型に切り替える。

**影響範囲**:
- [`master-sheet-schema.md`](./master-sheet-schema.md) — § 9〜11 を追記
- [`documents/README.md`](./documents/README.md) — 全面書き直し（書類一覧の SoT はシート側に）
- 各書類のプレースホルダ案 — 取引先/代理人を `{{買主会社名}}` `{{代理人氏名}}` 等のキーで差し込む方針に変更

### 2026-05-27: 「1案件=1スプレッドシート」から「1案件=1行+1フォルダ」に変更

**旧 North Star**（採用しない）:
- 1 案件ごとに案件サマリースプレッドシートを Drive に自動生成（テンプレからコピー）
- 案件マスタは集計用に主要列を冗長転記する Source of Truth
- 案件ごとのスプレッドシートで個別編集 → onEdit で マスタへ自動転記

**新 North Star**（こちらに移行）:
- 1 案件 = 1 行 + 1 フォルダ
- 案件マスタ 1 シートにすべての情報を集約（冗長転記レイヤなし）
- 行頭チェック + 書類列チェックで複数案件×複数書類を一括出力
- パターン選択で書類セットの初期チェックを自動投入
- ステータス: `未着手 / 出力済み / 完了 / 保留`（プルダウン）

**変更理由**:
- 月末に書類が集中する運用に対して、案件ごとシート方式では「シート間を行き来する手間」「集計のための転記レイヤ」「並列編集の競合」など実運用上のオーバーヘッドが大きい
- 集約型なら QUERY/SUMIFS で直接集計でき、データレイヤが 1 段減ってシンプル
- バッチ型出力（月末まとめ）への適合性が高い
- 営業がシート 1 枚を見れば全案件を俯瞰できる UX が良い

**変更により廃止される概念**:
- 案件サマリースプレッドシート（旧 § 7 で定義していた構造）
- onEdit によるマスタへの自動転記
- 「案件マスタは冗長転記先」という位置づけ → 「案件マスタは Source of Truth」に変更

**影響範囲**:
- [`master-sheet-schema.md`](./master-sheet-schema.md) — 全面書き直し
- [`phase1-user-tasks.md`](./phase1-user-tasks.md) — Phase 2 で実施するマスタ列追加作業の準備指示を追記
- [`../../../docs/schedule.md`](../../../docs/schedule.md) — Phase 2 以降の内容を新方針に再構成
- [`placeholder-rules.md`](./placeholder-rules.md) — 大きな変更なし（マスタ列名が変わる程度）
