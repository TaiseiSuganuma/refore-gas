# document-generator 仕様書

リフォレの仲介案件における書類作成業務を、フォーム入力からPDF出力まで自動化するシステム。
ゼロベースで設計された **目標形（North Star）** を最終的なゴールとし、段階的に近づける。

- **対象 GAS 種別**: Sheets コンテナバインド型（案件サマリーシート側）+ Forms トリガー（Phase 4 以降）
- **タイムゾーン**: `Asia/Tokyo`
- **GAS ランタイム**: V8

---

## 1. North Star（目標形）

```
┌─────────────────────────────────────────────────────────┐
│ ① 新規案件フォーム                                          │
│   - 売主情報、土地情報、立木情報、金額条件、契約日 等を入力 │

                       │ フォーム送信トリガー
                       ↓
┌─────────────────────────────────────────────────────────┐

│   - メニュー「書類出力」                                    │
```

## 2. 業務フローの理解

```
[地主] ─(A: 仕入)─→ [リフォレ] ─(B: 売上)─→ [買主 / 伐採業者]
```

契約書は A 系統（リフォレが甲＝買主）と B 系統（リフォレが甲＝売主）の2系統。
さらに「土地が絡むか / 立木のみか」で分岐。

| 系統 | パターン | 対象 | 契約書 |
|---|---|---|---|
| A | A① | 立木のみ | 立木売買契約書 |
| A | A② | 立木+土地 | 立木付土地売買契約書 |
| A | A③ | 土地のみ | 土地売買契約書 |
| B | B① | 立木のみ | お客様契約書_立木 |
| B | B② | 立木+土地（A③ 土地のみケースも含む） | お客様契約書_土地込 |

土地が絡む場合（A②/A③/B②）は法務局関連書類5種類が追加で必要。

## 3. 段階的アプローチ（Phase 設計）

各 Phase は **使える状態で止める**。Phase 完了時に事務担当が一部でも使える価値を提供する。

| Phase | ゴール | 状態 |
|---|---|---|
| **0** | 情報収集・仕様化 | 完了間近 |
| **1** | 既存「案件一覧」シートから「土地売買契約書」1書類をPDF出力する MVP | 着手予定 |
| **2** | 同シートから複数書類同時出力、パターン → 書類セット自動判定 | 未着手 |
| **3** | 明細繰返し対応、A①/A②/B①/B② の3契約書実装 | 未着手 |
| **4** | フォーム → 案件サマリーシート自動生成、案件マスタ追記 | 未着手 |
| **5** | フォームに登記簿PDF添付欄追加、registry-ocr と統合 | 未着手 |
| **6** | 見積書/請求書/法務局関連書類/社内シートを順次対応 | 未着手 |
| **7+** | 運用フィードバックを反映した改善 | 未着手 |

> **重要**: Phase 1〜3 は「現状の案件マスタ」を使い続ける。
> Phase 4 で「案件サマリーシート構造」に再設計するが、移行手順は Phase 4 着手時に確定する。

### Phase 1 詳細（最初の MVP）

**ゴール**: 既存のスプレッドシート（[【リフォレ】書類自動生成](https://docs.google.com/spreadsheets/d/12BEUxTqnADqWsmffGWos0tFe08FohEE-0RQJbLNAnEk/edit)）の案件一覧シートからアクティブ行を選び、土地売買契約書1書類をPDF出力できる状態を作る。

**手順**:
1. 「土地売買契約書」テンプレ（[fileId: 1AsZLK9_rmIF-VC62I42QMAmPmzNz0kp1KmegOZ6ApJI](https://docs.google.com/document/d/1AsZLK9_rmIF-VC62I42QMAmPmzNz0kp1KmegOZ6ApJI/edit)）に `{{key}}` プレースホルダを挿入
2. 案件一覧シートから案件IDをキーに該当行のデータを取得
3. プレースホルダ置換 → 指定フォルダにPDF保存
4. メニュー「書類出力」→「土地売買契約書」項目で実行
5. `docs/documents/purchase_contract_land.md` を仕様書として完成

**Phase 1 で意図的にやらないこと**:
- 複数書類対応 → Phase 2
- 案件パターン自動判定 → Phase 2
- 明細表の繰返し処理 → Phase 3
- フォーム連携 → Phase 4
- OCR統合 → Phase 5

### Phase 2 詳細（複数書類 + パターン判定）

**ゴール**: 案件パターン（A①〜B②）を案件一覧の列で判定し、必要な書類セットを自動選択。複数書類を同時にPDF出力。

**追加実装**:
- 案件一覧シートに「案件パターン」列を追加
- 書類マスタシートを新設（書類ID、テンプレID、対象パターン、明細含むか、命名規則）
- ダイアログUIで書類を複数選択（パターンに応じた書類が事前チェック済み）
- 各書類について順次 PDF 生成

### Phase 4 詳細（フォーム → サマリーシート）

**ゴール**: 新規案件をフォームから入力。送信トリガーで案件サマリーシートが自動生成され、Driveに案件フォルダが作られる。

**追加実装**:
- Google Form を作成（または GAS の Web App でカスタムフォーム）
- フォーム送信トリガー (`onFormSubmit`) で:
  - 案件ID発番
  - 案件フォルダ作成
  - テンプレから案件サマリーシートをコピー
  - フォーム入力値をシートに転記
  - 案件マスタに1行追記（サマリーシートへのリンク付き）
- 案件サマリーシート側にも document-generator のメニューを移植（コンテナバインド GAS）

### Phase 5 詳細（OCR 統合）

**ゴール**: フォームに登記簿PDF添付欄を作り、送信時に registry-ocr のロジックで自動抽出。結果を案件サマリーシートに自動投入。

**追加実装**:
- フォームに `登記簿PDF` 添付欄追加
- フォーム送信トリガーで添付PDFを取得 → registry-ocr の OCR ロジックを呼び出し
- 抽出結果（所在・地番・地目・地積・所有者氏名）を案件サマリーシートに投入
- registry-ocr 側のコードを `shared/` などに切り出し、両プロジェクトで共有する設計に変更検討

## 8. エラーハンドリング方針

- マスタに必須値が無い → 生成中断 + 不足列を表示
- テンプレのプレースホルダがマスタに無い → 警告ログ + 処理継続（未差込のままPDF出力）
- 一時 Docs はエラー時も `finally` で削除

---

## 9. ログ

- `Logger_` 名前空間 → Stackdriver
- 各書類の生成開始・終了・PDFリンクを記録
